# Bounding Boxes Documentation

## Overview
Geographic bounding boxes are a crucial component of our data filtering pipeline. Because mass shootings are localized events, we aim to isolate Twitter discussions originating specifically from or around the affected communities to gauge localized sentiment and reactions. 

The pipeline filters tweets by checking whether their geographic coordinates (`latitude`, `longitude`) fall within a rectangular bounding box that encompasses the incident city.

## Collection Methodology
The bounding boxes are collected programmatically using the OpenStreetMap (OSM) **Nominatim API** via direct HTTP requests. The pipeline retrieves **full city polygons** (when available) and computes bounding boxes from those geometries, ensuring coverage of the entire city area.

The function queries the API with the following city names:
1. **Uvalde, TX** (Incident Date: May 24, 2022)
2. **Highland Park, IL** (Incident Date: July 4, 2022)
3. **Monterey Park, CA** (Incident Date: January 21, 2023)
4. **Nashville, TN** (Incident Date: March 27, 2023)

### Key Features
- **Polygon-aware**: Requests `polygon_geojson=1` from Nominatim to retrieve full city boundaries, not just simple rectangular bboxes.
- **Buffer expansion**: Supports an optional `buffer_km` parameter to expand the bounding box outward (useful for capturing tweets from surrounding areas or accounting for geocoding imprecision).
- **Source tracking**: Records whether each bbox was derived from a polygon boundary or a simple boundingbox, stored in the `source` field.
- **Nominatim compliance**: Accepts an `email` parameter to include in the User-Agent header per Nominatim usage policy.

## Data Storage Format
The collected bounding boxes are stored as a JSON object in `data/bboxes.json`. Each bbox includes lat/lon bounds and a source indicator. Format:

```json
{
    "Uvalde, TX": {
        "lat_min": 29.1738,
        "lat_max": 29.2435,
        "lon_min": -99.8143,
        "lon_max": -99.7426,
        "source": "polygon"
    },
    "Highland Park, IL": {
        "lat_min": 42.1539,
        "lat_max": 42.2037,
        "lon_min": -87.8384,
        "lon_max": -87.7813,
        "source": "polygon"
    }
}
```

*(Note: The coordinates above are illustrative examples. Actual coordinates are dynamically generated by the `collect_bounding_boxes()` function.)*

### Source Field
- **`"polygon"`**: Bounding box computed from the full city polygon returned by Nominatim.
- **`"bbox"`**: Fallback; simple bounding box returned by Nominatim (used if polygon is unavailable).

## Function Signature & Usage

The `collect_bounding_boxes()` function is defined in the `notebooks/data_processing_pipeline.ipynb` notebook:

```python
def collect_bounding_boxes(locations, output_file=BBOX_FILE, buffer_km=0, email=None):
    """
    Collect city-wide bounding boxes using Nominatim polygon_geojson.
    
    Args:
      locations: iterable of location name strings
      output_file: path to write JSON with bounding box data
      buffer_km: expand each bbox by this many kilometers (approx)
      email: contact email to include in User-Agent (Nominatim policy)
    
    Returns:
      dict of bboxes keyed by location name
    """
```

### Parameters

- **`locations`**: List of city names to query (e.g., `["Uvalde, TX", "Nashville, TN"]`).
- **`output_file`**: Path where results are saved (default: `../data/bboxes.json`).
- **`buffer_km`**: Distance in kilometers to expand the bbox on all sides (default: `0`—no expansion).
  - `0`: Exact city boundary from Nominatim.
  - `5` or higher: Includes surrounding areas (useful for regional context or handling geocoding imprecision).
- **`email`**: Your contact email (recommended for Nominatim API compliance; improves rate limits).

### Example Usage

```python
# Collect bboxes for incident cities with 5 km buffer
BBOXES = collect_bounding_boxes(
    list(INCIDENTS.keys()), 
    buffer_km=5, 
    email="your_email@example.com"
)

# Collect exact city boundaries (no buffer)
BBOXES = collect_bounding_boxes(
    list(INCIDENTS.keys()), 
    buffer_km=0
)
```

## How It Works in the Pipeline
During the location and date filtering step (Section 3 of the notebook), the pipeline performs the following steps for each tweet:
1. Loads the tweet's `latitude` and `longitude`.
2. Iterates through the predefined list of incident cities and their precomputed bboxes.
3. Checks if the tweet's date falls within the $\pm 2$ day window of the incident.
4. If the date matches, it applies a geographic mask:
   ```python
   lat_mask = (df['latitude'] >= bbox['lat_min']) & (df['latitude'] <= bbox['lat_max'])
   lon_mask = (df['longitude'] >= bbox['lon_min']) & (df['longitude'] <= bbox['lon_max'])
   ```
5. Tweets satisfying both temporal and spatial constraints are tagged with their respective `matched_incident` name and retained for keyword filtering.
